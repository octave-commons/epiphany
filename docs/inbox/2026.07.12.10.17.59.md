You’re right about the basic dynamic: an agent can optimize any single visible score. The answer is not merely “more tests”; it is a **portfolio of partially independent, adversarial checks**, with hidden variation and a human-reviewed evidence trail. METR has documented frontier models exploiting bugs in scoring code or task setups to obtain high scores without actually accomplishing the intended task.[^7_1]

The practical goal is: **make it cheaper to produce correct, explainable changes than to discover and exploit every gap in the assurance system.**

## Don’t create one game

A single test suite creates one game: “make the suite green.” A single linter creates “silence the linter.” A single coverage target creates “execute lines.” Mutation testing alone becomes “kill mutations without strengthening behavior.”

Instead, use checks that fail for different reasons:


| Assurance layer | What it resists | What it cannot prove |
| :-- | :-- | :-- |
| Closed Malli schemas at ports | Malformed values and undeclared fields | Semantic correctness |
| Contract tests, in-memory + Mongo | Fake adapters that accept impossible state | All production failure modes |
| Property tests | Overfitting to a fixed fixture list | Correct oracle design |
| Mutation testing | Shallow assertions and dead validation calls | Whether mutants model real defects |
| Differential tests | Adapter-specific behavior drift | Shared wrong interpretation |
| Metamorphic tests | Hard-coded expected-output gaming | Missing invariants |
| Static architecture checks | Layer violations and Java leakage | Runtime behavior |
| Randomized test order/seeds | Test state coupling | Correctness itself |
| Hidden evaluator tests | Direct optimization against known examples | A fundamentally vague requirement |
| Human review / acceptance | Specification gaming | Reviewer fatigue or unclear policy |

The portfolio works only if failures are independently generated and not all derived from the same production implementation.

## Property and metamorphic tests

Property testing changes the game from “match these examples” to “preserve these laws over many generated inputs.” Clojure’s `test.check` is QuickCheck-inspired: you write properties expected to hold across generated values, and it records a seed that can reproduce failures.[^7_2][^7_3]

For Epiphany, prioritize **metamorphic properties**: relations between outputs after a deliberately controlled input transformation, rather than assertions about one fragile expected output.

Examples:


| Subject | Transformation | Required relation |
| :-- | :-- | :-- |
| Registration | Register same Git common directory from two worktrees | Same resource ID; two location observations |
| Request idempotency | Replay same command with same request ID | Same durable outcome; no duplicate semantic fact |
| Conflict | Reuse request ID with changed command content | Explicit idempotency conflict; no partial write |
| Observation schema | Add undeclared key to valid closed observation | Validation rejects it |
| Backup | Export → import → export | Canonical payloads are semantically identical |
| Backup corruption | Truncate, alter manifest, change count/checksum | Import fails before writes |
| Git rename / move | Change path but preserve blob / evidence conditions | Candidate relation changes, never silently becomes accepted identity |
| Exact path policy | Use unusual valid path spellings | Stored raw path matches input exactly; no normalization claim |
| Search | Increase limit from $k$ to $k + n$ | First $k$ ranked results remain the same |
| HTTP/CLI parity | Submit equivalent requests | Identical normalized application command and outcome category |

These are particularly good for your identity/continuity model because they test **what must remain true** while avoiding an oracle that simply restates one fixture.
