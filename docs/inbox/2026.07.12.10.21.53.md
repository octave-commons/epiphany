## What PROCESS.md should become

I agree that we should start there. It should become the canonical document for **how Epiphany creates, validates, accepts, revises, and retires knowledge-bearing work**—not a second copy of every board transition and not a bag of general productivity advice.

Its job is to answer:

> Given an uncertainty, change request, or discovery, what artifacts must exist, what claims may they make, what evidence qualifies them, who/what may promote them, and how is the resulting work accepted or revised?

The new `PROCESS.md` should contain:

1. **Purpose, scope, and non-goals** — process governance, not coding style, board CLI reference, or product architecture.
2. **Authority and conflict order** — README informs purpose; ADRs make architecture policy; PROCESS governs artifact lifecycle; templates/checker define document contracts; board docs define board mechanics.
3. **Epistemic vocabulary** — observed source, derived finding, provisional proposal, accepted decision; document versus structured record versus generated view.
4. **Lifecycle** — intake -> research brief -> research activity -> findings/report -> design -> ADR when necessary -> board planning -> implementation -> review -> acceptance -> retrospective.
5. **Artifact gates** — minimum requirements for research, design, ADR, story, implementation evidence, and exceptions.
6. **Research governance** — explicitly define what an act of research and a research artifact are, rather than treating research as prose.
7. **Board integration** — link to the one canonical Kanban policy; do not repeat its state graph inside PROCESS.
8. **Exceptions** — structured, scoped, owned, time-bound, reviewable, and expiring; never an undocumented waiver.
9. **Governance changes** — how templates, rules, checker behavior, and process itself evolve.
10. **Documentation quality rollout** — templates/exemplars first, structural checker next, semantic warnings then ratcheted enforcement.

## What should move out

The rewrite should deliberately remove or delegate:

- Detailed Kanban FSM diagrams and transition commands -> the canonical board policy.
- Concrete test and static-analysis commands -> `AGENTS.md` or a generated developer-command reference.
- Namespace and Clojure construction rules -> a reconciled `STYLE.md`.
- Architecture decisions -> ADRs.
- Detailed per-document fields -> future `docs/standards/document-kinds/*.md` contracts.

That makes PROCESS durable. It governs **why and when** things happen; specialized documents govern **how**.
