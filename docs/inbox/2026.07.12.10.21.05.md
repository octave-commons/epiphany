```md
# ADR-004: Contract-First Adversarial Verification

## Context

Epiphany is a Git-backed knowledge system whose durable records distinguish
observed facts, derived outputs, provisional candidates, and explicitly
accepted decisions. Incorrect persistence, silent schema bypass, or a false
"no evidence" result can create misleading continuity claims.

The system has multiple adapter profiles. In-memory adapters make unit tests
fast, but permissive doubles can accept records that production persistence
rejects. A green unit suite is therefore insufficient evidence that commands
produce valid durable observations.

AI agents and conventional developers can optimize visible checks without
satisfying the intended behavior. Coverage, fixed examples, lint cleanliness,
mutation score, and one adapter's behavior are individually gameable.

## Decision

Epiphany adopts a contract-first, multi-oracle verification architecture.

1. Every externally received command, durable observation, persisted document
   decode, backup import, and public result has a named versioned schema.

2. Schema enforcement is mandatory at application and adapter boundaries.
   Validation may be wrapped at composition time, but no public persistence
   operation may rely solely on a caller remembering to validate.

3. The in-memory adapter is a contract-enforcing reference adapter. It is not
   allowed to accept states that the durable adapter rejects under the shared
   port contract.

4. Every adapter implementation must pass shared contract tests. Adapter
   integration tests remain necessary for implementation-specific behavior such
   as BSON encoding, indexes, filesystem failure, Git object access, and
   service availability.

5. Correctness is established through a portfolio of independent evidence:
   schema validation, static analysis, example tests, property and metamorphic
   tests, adapter differential tests, targeted mutation testing, and
   integration tests.

6. No aggregate score substitutes for a hard integrity gate. In particular,
   invalid records must not persist; unknown schema versions must not be
   interpreted as known versions; unavailable, corrupt, empty, and
   not-implemented outcomes must remain distinguishable.

7. CI-generated evidence is authoritative over agent-authored completion
   claims. Required checks, random seeds, mutation reports, and adapter
   contract results must be generated or verified by CI.

8. Agent-visible checks may be supplemented by private or rotated evaluation
   cases. Stable laws remain public; their concrete inputs, seeds, fixtures,
   and adversarial variants may vary.

## Consequences

### Positive

- Unit tests and production adapters share an explicit behavioral contract.
- Schema failures occur near the responsible boundary instead of late in a
  Mongo, Lucene, HTTP, or backup path.
- The project measures assurance as multiple corroborating observations rather
  than one green test result.
- New persistence operations require an explicit schema and contract-test
  registration.
- The system better preserves its epistemic rule: missing, malformed, and
  unavailable evidence cannot silently become negative evidence or identity.

### Costs

- More test infrastructure, slower integration and mutation jobs, and more
  deliberate fixture/version maintenance.
- Some developer friction: a new command, observation, or adapter operation
  requires schema, validator, reference behavior, and tests before it is
  usable.
- Private/rotated checks require CI secret handling and careful reproducibility
  mechanisms.
- Independent oracles must not all be derived from the same implementation.

## Non-decisions

This ADR does not:

- Select a permanent linter, mutation-testing engine, or property-testing
  library.
- Require a specific static type system across all Clojure namespaces.
- Require zero Java interop globally; interop remains permitted in explicitly
  infrastructure-owned adapters.
- Define domain schemas themselves or individual schema migrations.
- Replace explicit human acceptance of identity and continuity decisions.

## Required invariants

- A rejected write leaves adapter state unchanged.
- Every public persistence write operation has a named versioned input schema.
- Every read/decode validates the schema version it claims to represent.
- In-memory and durable adapters agree on acceptance/rejection and
  domain-level failure categories for their shared contract.
- A candidate relation, a missing source, an unavailable service, and an empty
  result remain distinct outcomes.
- A passing quality metric cannot waive a failed integrity gate.

## Alternatives considered

### Trust production-only validation

Rejected. It allows fast unit suites to certify behavior that cannot persist.

### Make the in-memory adapter emulate Mongo exactly

Rejected. The reference adapter must enforce shared semantics, but reproducing
Mongo internals would create a second fragile database implementation.

### One comprehensive end-to-end suite

Rejected. It is slow, hard to diagnose, and still vulnerable to a single
incorrect oracle or fixture set.

### Coverage threshold as the primary gate

Rejected. Executed code is not evidence that relevant assertions or
epistemically correct outcomes were verified.

### One quality score

Rejected. Weighted scores allow a serious integrity failure to be offset by
superficial wins elsewhere.

## Follow-up design work

- Define schema-boundary wrappers and the operation-to-schema registry.
- Define the adapter contract-test harness.
- Define the CI assurance matrix and required/private check separation.
- Define the Java-interop inventory and layer-boundary checker.
- Select and trial static-analysis, property-testing, and mutation-testing
  tooling.
- Define a machine-readable evidence artifact generated by CI.
```


## Keep the ADR narrow

The ADR should establish these **non-negotiable policies**:

- Schemas are enforced facts, not documentation.
- The in-memory adapter is a reference contract implementation, not a bag of atoms.
- No one metric or test run is authoritative.
- Checks must be partially independent.
- Human acceptance remains required where the domain makes epistemic claims.

The following should stay out of the ADR and go into `docs/designs/`:

- Whether Splint is adopted, its rule configuration, and ratchet thresholds.
- The chosen mutation tool and exact score target.
- Exact clj-kondo rules.
- CI YAML and schedules.
- Specific private-test fixtures and seeds.
- The initial count of allowed Java interop calls.

Those are implementation decisions likely to evolve; embedding them in an ADR would make an enduring architectural rule brittle.

# @GitHub continue

I drafted **ADR-004: Contract-First Adversarial Verification** and left it as `draft`, appropriately: it makes a real architectural commitment but does not prematurely lock the project into a particular linter, mutation engine, or CI vendor. It formalizes the core policy that correctness must be established by corroborating, partially independent evidence—not a single green suite or agent report.

## What it decides

The ADR requires named, versioned contracts for boundary data; mandatory validation at application and adapter boundaries; a contract-enforcing in-memory reference adapter; common adapter-law tests; and hard integrity gates that cannot be outweighed by aggregate quality scores.

It explicitly preserves the Epiphany-specific epistemic distinctions: invalid, unavailable, corrupt, unknown, empty, provisional, and accepted must remain different states. Similarity, a passing test, or an unavailable source must never silently become durable identity.

## What it leaves open

It intentionally does **not** decide:

- Splint versus another style linter.
- Mutant, Heretic, or another mutation-testing tool.
- A project-wide static type system.
- Exact CI topology, private test fixtures, or score thresholds.
- A zero-Java policy; instead, it permits Java in dedicated infrastructure adapters while making layer boundaries a follow-up concern.

That keeps the ADR durable while allowing the implementation strategy to be tested and adjusted.

# @GitHub continue

I added the implementation-level companion design, **Verification Architecture**. It translates ADR-004 into boundaries, contracts, independent evidence sources, CI jobs, and small board-sized implementation slices.

## Design decisions made concrete

The design specifies:

- A single operation-to-schema registry, with no permissive fallback for unregistered public writes.
- A shared validation gateway plus composition-time validating-port wrappers and adapter-local defense in depth.
- An in-memory adapter that acts as a contract-enforcing reference implementation, without attempting to reimplement Mongo internals.
- A reusable adapter law harness run against both in-memory and Mongo implementations.
- Read/decode and backup-import validation—not only write validation.
- Injected clock/ID capabilities so domain decisions are deterministic and do not need direct JVM `Date`/UUID construction.
- CLI/HTTP parity through a common normalized command contract.
- Java-interop inventory and ratchet checks by architectural layer.
- Targeted mutation testing against schema gateways and boundary behavior, rather than a meaningless whole-repo percentage.
- A CI-generated evidence artifact that records observed checks, seeds, adapter runs, mutation outcomes, and interop deltas.


## Implementation order

The document breaks the work into ten slices:

1. Schema operation registry.
2. Validating port wrapper.
3. In-memory reference adapter parity.
4. Mongo contract alignment.
5. Read and import validation.
6. Shared CLI/HTTP command contract.
7. Static architecture boundary gate.
8. Generative and metamorphic laws.
9. Mutation-testing pilot.
10. CI evidence artifact and protected rotated evaluations.

That order respects the project’s “schemas/contracts precede adapters” rule and creates a meaningful green baseline before adding more elaborate tooling.
