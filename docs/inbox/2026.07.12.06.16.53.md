You already have the beginnings of a strong schema system—versioned, closed Malli maps in `epiphany.law.observation` and a shared registry—but enforcement is inconsistent. The key is to make schema validation an unavoidable property of every boundary and every test adapter, rather than an assertion individual tests opt into. 

## Current gap

The observation schemas are thoughtfully strict: they use closed maps, versioned schema names, required provenance envelopes, typed UUIDs, exact-path representation, and explicit observation types. 

But the in-memory observations adapter accepts values with simple `swap!` calls, while Mongo explicitly validates only repository-location observations before writing. That means other persisted record kinds can be accepted by unit tests without being checked against their law schemas. 

## One validation gateway

Create a single registry-backed validation namespace, and ban direct storage through the port without passing it.

```clojure
(ns epiphany.application.validate
  (:require [epiphany.law.registry :as registry]))

(defn valid!
  [schema-name value]
  (if (registry/valid? schema-name value)
    value
    (throw
     (ex-info "Schema validation failed"
              {:code :schema-validation-failed
               :schema/name schema-name
               :schema/explain (registry/explain schema-name value)
               :value value}))))
```

Avoid embedding a large invalid value in production exception data if it could contain repository content or secrets; log a redacted diagnostic separately.

Then make record functions validate at the port’s write boundary:

```clojure
(defn record-ingestion-run!
  [store observation]
  (validate/valid! "observation/ingestion-run-v1" observation)
  (swap! store conj observation)
  nil)
```

The **same gateway** must be used by:

- In-memory observations adapter.
- Mongo observations adapter.
- Backup import.
- Fixture loader.
- Any future external ingestion adapter.

This eliminates the current “valid only in Mongo” split.

## Map operation to schema

Do not infer a schema only from the map’s `:observation/type`; make the mapping explicit and complete. This forces a code review when a new persisted record type is introduced.

```clojure
(def operation->schema
  {:record-repository-location!
   "observation/repository-location-v1"

   :record-ingestion-run!
   "observation/ingestion-run-v1"

   :record-checkpoint!
   "observation/projection-checkpoint-v1"

   :record-revision-at-path!
   "observation/revision-at-path-v1"

   :record-section-extraction!
   "observation/section-extraction-v1"})

(defn validate-write!
  [operation observation]
  (if-let [schema-name (get operation->schema operation)]
    (valid! schema-name observation)
    (throw (ex-info "Write operation has no registered schema"
                    {:code :unregistered-write-schema
                     :operation operation}))))
```

This is more trustworthy than a permissive fallback. A new port operation should fail loudly until it has a declared data contract.

## Wrap all adapters

Rather than relying on every implementation author to remember a validation call, provide a decorator that wraps the observations port at composition time:

```clojure
(defn validating-observations-port
  [observations]
  (-> observations
      (update :record-repository-location!
              (fn [record!]
                (fn [observation]
                  (validate-write! :record-repository-location! observation)
                  (record! observation))))
      (update :record-ingestion-run!
              (fn [record!]
                (fn [observation]
                  (validate-write! :record-ingestion-run! observation)
                  (record! observation))))
      (update :record-checkpoint!
              (fn [record!]
                (fn [observation]
                  (validate-write! :record-checkpoint! observation)
                  (record! observation))))
      (update :record-revision-at-path!
              (fn [record!]
                (fn [observation]
                  (validate-write! :record-revision-at-path! observation)
                  (record! observation))))
      (update :record-section-extraction!
              (fn [record!]
                (fn [observation]
                  (validate-write! :record-section-extraction! observation)
                  (record! observation))))))
```

Composition should use it for **both** local/in-memory and services/Mongo profiles:

```clojure
{:observations
 (validating-observations-port raw-observations)}
```

Keep validation in the Mongo adapter too as defense in depth; the decorator protects normal application flow, while adapter-local validation protects scripts, direct REPL use, and future callers.

## Validate read boundaries too

Write validation alone is not enough. Mongo document decoding, backup imports, Git adapters, and HTTP decoders can introduce malformed maps.

Require this pattern:

```clojure
(defn decode-repository-location
  [document]
  (-> document
      mongo-document->observation
      (validate/valid! "observation/repository-location-v1")))
```

For every read, distinguish:

- **Observed source failure:** Git object missing, database unavailable, network timeout.
- **Data corruption / contract failure:** a record exists but cannot decode or violates its claimed schema version.

The latter should be a named, reviewable integrity finding—not silently omitted, defaulted, or converted into an empty result. That is especially important for Epiphany because empty evidence must not be confused with unavailable or malformed evidence.

## Make schemas coverage-complete

Add a test that fails if any persisted observation type lacks an enforcement mapping. The schemas already have distinct observation type/version definitions in the law registry, so use that registry as the authoritative inventory. 

```clojure
(deftest every-persisted-observation-schema-is-enforced
  (let [schemas (registry/observation-schema-names)
        enforced (set (vals validate/operation->schema))]
    (is (= schemas enforced)
        {:missing (sort (clojure.set/difference schemas enforced))
         :orphaned (sort (clojure.set/difference enforced schemas))})))
```

Better still, represent the relationship as data:

```clojure
{:operation :record-ingestion-run!
 :schema "observation/ingestion-run-v1"
 :observation-type :ingestion/run-completed
 :persistence :append-only}
```

Generate decorators, contract tests, documentation, and storage mappings from this registry. This prevents separate hand-maintained lists from drifting.

## Add adapter contract tests

Every observations implementation should be passed through the same law suite. In-memory and Mongo must agree on schema rejections, idempotency, and returned error codes—not necessarily raw exception messages or driver exceptions.

```clojure
(defn observations-contract-tests
  [{:keys [make-port]}]
  (testing "rejects an invalid record before persistence"
    (let [port (make-port)
          bad {:observation/type :ingestion/run-completed}]
      (is (thrown-with-msg?
           clojure.lang.ExceptionInfo
           #"Schema validation failed"
           ((:record-ingestion-run! port) bad)))))

  (testing "does not persist rejected data"
    ;; Attempt invalid write, then query/export port state;
    ;; assert that there is no newly recorded observation.
    ))
```

Run the exact suite for:

- Reference in-memory adapter on every unit-test run.
- Mongo adapter against an ephemeral Mongo instance.
- Any future database or remote-service adapter.

Your existing in-memory test checks that the adapter map itself satisfies the **port-shape schema**. That is useful, but it only verifies that required keys are functions; it does not prove those functions honor the data contract. 

## Enforce at construction time

The current `application-ports-schema` verifies that an adapter set has the expected keys and function-shaped entries. Strengthen it in two separate ways:

1. Validate the **shape** of the adapter map at composition time.
2. Wrap every port function with input/output validation at composition time in development and tests.

For example, a generic guarded port wrapper can verify that every return value conforming to a declared schema is valid, and every command passed in is valid. For functions whose return values are `nil`, validate their input; for queries, validate both request and response.

```clojure
(defn guarded
  [input-schema output-schema f]
  (fn [input]
    (valid! input-schema input)
    (let [output (f input)]
      (valid! output-schema output))))
```

Use named command and result schemas rather than anonymous maps so failures point to stable contracts.

## Property-test schemas

Example fixtures catch only known mistakes. Add generative tests that produce valid records, mutate one field at a time, and prove the validator rejects invalid variants.

Important laws:

- A valid closed observation becomes invalid when an undeclared key is added.
- Replacing a UUID with a string is invalid.
- Replacing `:path/comparison :exact` with any other value is invalid.
- A record’s declared `:observation/schema-version` must match the version in the registered schema name.
- A read/decode round trip preserves the canonical observation map.
- Invalid writes do not alter adapter state.
- Idempotent retries return/retain the same recorded fact; same request ID with materially different content fails as a conflict.

Use Malli generators for schemas where practical, but hand-curate generators for identity and Git evidence types if their semantic constraints exceed structural shape.

## Schema version enforcement

Schemas are versioned in their registry name and observations carry `:observation/schema-version`, but add a direct invariant that ties them together. The record must not merely have a positive integer—it must state the version expected by the storage operation.

```clojure
(def expected-schema-version
  {"observation/repository-location-v1" 1
   "observation/ingestion-run-v1" 1
   "observation/projection-checkpoint-v1" 1
   "observation/revision-at-path-v1" 1})

(defn validate-version!
  [schema-name observation]
  (let [expected (get expected-schema-version schema-name)
        actual (:observation/schema-version observation)]
    (when-not (= expected actual)
      (throw (ex-info "Observation schema version mismatch"
                      {:code :schema-version-mismatch
                       :schema/name schema-name
                       :expected expected
                       :actual actual})))
    observation))
```

When you introduce v2, preserve `v1` as a decoder/validator for historical data, add a deliberate migration, and never reinterpret v1 documents as if they were v2.

## CI gates

Turn this into layers that cannot be bypassed accidentally:

```yaml
jobs:
  lint:
    steps:
      - run: clojure -M:lint

  unit-contract:
    steps:
      - run: clojure -M:unit-test

  mongo-contract:
    services:
      mongo:
        image: mongo:8
        ports: ["27017:27017"]
    steps:
      - run: clojure -M:integration-test

  schema-compatibility:
    steps:
      - run: clojure -M:schema-check
```

The repository’s current CI unit job runs `clojure -M:unit-test`; it should be retained for speed, but it cannot be the only required gate for persistence claims. 

A `:schema-check` alias should run:

- Registry compilation for every schema.
- Enforcement-registry completeness check.
- Fixture validation for every supported historical schema version.
- Adapter contract tests for in-memory implementations.
- Serialization compatibility checks against versioned fixtures.
- Static linting of namespaces that write persistent data, ensuring they use the validation gateway.

## Make bypasses visible

Finally, establish one hard rule:

> No code may call a raw persistence adapter from an application handler, CLI handler, HTTP handler, or domain namespace.

Only the composition root can receive raw adapters; it immediately wraps them in validated ports. In tests, expose both only deliberately:

- `raw-adapter` for adapter-specific mapping tests.
- `validated-port` for all application/domain tests.

This keeps schema enforcement factual and reviewable rather than dependent on team memory.
