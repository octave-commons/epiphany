You’re right. I framed pseudocode as a disclaimer—“Lisp-shaped process form”—when it should be treated as **first-class design data**.

The form does not need to compile today to be valid. Its value is that it gives the future system a shape it can grow into, test against, and progressively realize:

```clojure
(η Discovery
   (-> (-> Describe specify define)
       (-> μ shape extern domain infra))
   Π)
```

The repository currently has partial instances of this form. That is not nonconformance in the sense of invalidity; it is an **incomplete evaluation** of the form.

## Pseudocode is data

In a Lisp project, unexecuted code-shaped notation can be more precise than prose because it preserves:

- **Composition** — which concepts feed which others.
- **Grouping** — which operations form a coherent unit.
- **Order** — what has to be understood before a later transformation is trusted.
- **Open symbols** — what the system has not implemented or defined yet.
- **Future executability** — a path from design to actual functions, macros, data, checks, or traces.

The form is not pretending `η`, `Discovery`, `shape`, or `Π` are current vars. They are **declared symbols in the system’s future vocabulary**.

```clojure
;; Valid design data now:
(η Discovery
   (-> (-> Describe specify define)
       (-> μ shape extern domain infra))
   Π)

;; Potentially executable later:
(defn construct [η μ]
  ...)
```

The first constrains the second. If no one writes the pseudocode, the eventual implementation has no shared shape to conform to, challenge, or improve.

## “Valid data” is the point

Your statement gets at the critical distinction:

```text
Reader-valid data != compiler-valid program
```

A form can be valid as:

- A design artifact.
- An architectural constraint.
- A future interface.
- A test oracle.
- A macro target.
- A data model.
- A trace/provenance representation.
- A pedagogical expression of the system’s ontology.

It does not become meaningless because the runtime cannot evaluate it today.

For Epiphany especially, this is almost foundational: Markdown, EDN, Git history, cards, ADRs, and source code are all addressable artifacts. Their present executability is not the criterion for whether they are meaningful evidence or valid objects of analysis.

## The actual promise

The kernel should say something stronger than “existing code may not conform.”

It should say:

> The construction form is a declared target shape. Existing code may partially instantiate it, contradict it, or leave parts unimplemented. Those differences are observable architectural data: they create migration candidates, not grounds to weaken the form.

Then:

```text
Current source
  + construction form
  -> architectural observations
  -> bounded remediation/design work
  -> increasingly faithful implementation
```

That makes the kernel a living instrument for reading the repository, not merely a rulebook used to condemn it.

## Replace the migration language

The current `STYLE.md` migration section should be reframed like this:

```md
## The form is valid before it compiles

The construction form is valid design data whether or not every symbol is
implemented as executable Clojure today. It declares the target shape against
which existing code can be read.

Existing source may partially instantiate the form, contain older shapes, or
contradict a current boundary law. Each difference is an architectural
observation:

- A `domain.*` namespace that imports an SDK identifies an unresolved `extern.*`
  boundary.
- A persisted map without a `law.*` contract identifies an unexpressed
  admissibility rule.
- A driver response escaping an adapter identifies an unshaped foreign value.
- An `infra.*` callback making a policy choice identifies domain meaning that
  has not yet been given a home.

These observations are not grounds to erase or weaken the form. They are
candidates for discovery, design, and bounded remediation. The form becomes
more executable over time because actors preserve and work the gap between its
declared shape and the current repository.

Do not perform a repository-wide namespace shuffle merely to make paths look
conformant. Make each change as a reviewable realization of a named part of the
form, with contracts and verification appropriate to the boundary.
```


## A future use

Eventually, this can become literally inspectable data:

```clojure
(def construction-form
  '(η Discovery
      (-> (-> Describe specify define)
          (-> μ shape extern domain infra))
      Π))
```

Then tools can ask useful questions without reducing the form to a dumb folder checker:

```clojure
{:observation/kind :kernel-gap
 :observed/source 'epiphany.domain.ingestion
 :expected/layer :domain
 :observed/foreign-import 'org.eclipse.jgit.lib.Repository
 :implied/missing-boundary :extern
 :status :provisional}
```

The code need not compile for that expression to be valuable. The form is already a shared object that can shape reasoning, documentation, review, and eventually analysis.
