You’re right. I produced **administratively valid cards**, not sufficiently executable ones. They have scope and acceptance bullets, but they do not reliably specify the implementation boundary, interfaces, invariants, test oracle, failure behavior, non-goals, evidence required for completion, or how to detect a superficially green implementation. That is exactly the gap ADR-004 is trying to address—and the board cards should have met that standard themselves.

The root problem is document-kind drift: the repository has some frontmatter guidance, but no enforced contract for what an ADR, design, story, epic, research report, or process record must contain. The inventory also shows inconsistent metadata shapes across ADRs, designs, stories, epics, and research documents. A high-quality individual document does not solve that; authors imitate different precedents and the corpus drifts.

## Corrective principle

We need to treat documentation as a **typed, reviewable corpus**, not formatted prose.

```text
Document kind
  -> required metadata
  -> required sections
  -> required relations
  -> allowed status transitions
  -> quality rules
  -> examples and counterexamples
  -> automated checks
  -> human review rubric
```

Templates are the authoring interface. The checker is the enforcement mechanism. Good/bad examples teach judgment. Review remains necessary for semantic adequacy.

## Why templates alone fail

A template can still be gamed:

```md
## Risks
None.
```

```md
## Acceptance criteria
- Tests pass.
```

```md
## Dependencies
None.
```

Those satisfy headings while conveying no real implementation contract.

So each document type needs three layers:


| Layer | Purpose | Example |
| :-- | :-- | :-- |
| **Structural contract** | Required metadata, headings, links, section order | Every story has `Scope`, `Out of scope`, `Acceptance criteria`, `Verification`, and `Dependencies` |
| **Semantic quality rules** | Detect weak, contradictory, or incomplete claims | Acceptance criteria must name observable behavior and a verification method |
| **Exemplars and anti-exemplars** | Teach authors and reviewers what “good” means | A good implementation card versus a vague card that says “add validation” |

The checker can prove structural conformance and identify strong smells. It cannot prove that a task is well scoped or the right architecture; that remains an explicit human judgment.

## Document taxonomy

I would define a small stable set of document kinds first.


| Kind | Primary purpose | Must answer |
| :-- | :-- | :-- |
| ADR | Make one durable architectural policy decision | What was decided, why, consequences, alternatives, non-decisions, reversibility |
| Design | Turn accepted policy into an implementation model | Components, data/contracts, boundaries, failures, rollout, acceptance evidence, open decisions |
| Epic | State a user/outcome area and delivery boundary | Outcome, scope, success signals, excluded work, child-story strategy |
| Story / engineering slice | Define one executable change | Problem, exact scope, non-goals, interfaces/data affected, invariants, test plan, completion evidence |
| Research | Preserve sourced findings and uncertainty | Question, method, sources, findings, conflicts, limitations, decisions informed |
| Process | Define repeatable team/agent workflow | Trigger, actors, steps, inputs/outputs, exceptions, enforcement, examples |
| Planning record | Capture a pending decision or triage outcome | Decision needed, options, evidence, owner, deadline/exit condition |
| Example / fixture | Teach or test a document contract | Classification, why good/bad, specific rules illustrated |

Do not make every Markdown file one of these. Inbox, notes, archived imports, and rough working material should stay intentionally lightweight and be excluded from strict checks.

## Story template needs more meat

For an engineering story, I would require this—not merely `Scope` and acceptance criteria:

```md
---
id: "..."
title: "ENG-017A: Define the schema operation registry"
kind: story
status: incoming
priority: P0
phase: 1
epic: "..."
design: "docs/designs/verification-architecture.md"
points: 3
dependency: ["..."]
verification: ["unit-contract", "schema-check"]
risk: medium
---

# ENG-017A: Define the schema operation registry

## Intent

What defect or capability this slice changes, stated in operational terms.

## Decision context

The specific design section and ADR rule this work implements.

## Scope

Concrete deliverables, named namespaces/files where known, and boundaries of
responsibility.

## Non-goals

Explicit work that this slice must not absorb.

## Contract changes

Input/output data, schemas, ports, CLI/HTTP behavior, persistence semantics,
and compatibility impact. State `None` explicitly if none applies.

## Invariants

- An unregistered public persistence operation fails explicitly.
- A schema-version mismatch is rejected before mutation.
- Validation errors do not expose raw repository content.

## Implementation outline

The intended seams and sequence—not a full solution, but enough to prevent a
vague implementation.

## Verification

| Claim | Evidence | Command or test location |
|---|---|---|
| Every operation has one schema mapping | Registry completeness test | `...` |
| Unknown operation fails | Negative unit test | `...` |
| Invalid value is rejected | Schema-law test | `...` |

## Acceptance criteria

Observable outcomes, each traceable to verification above.

## Dependencies and interfaces

What must already exist; what later cards may rely on.

## Risks and open questions

Known uncertainty. `None` is an explicit claim requiring review.

## Completion evidence

Required artifacts, test output, migration notes, and card-comment content.
```

This makes a story executable without pretending it is a design document. It also makes “tests pass” insufficient: each acceptance criterion must link to a named claim and an independent verification method.

## Design template

The verification design I created has much of the needed substance, but it should conform to a single design template rather than rely on my judgment. It should require:

```text
Purpose
Decision basis and authoritative inputs
Terminology / domain model
Current state and problem
Goals and non-goals
Constraints and assumptions
Architecture / component responsibilities
Data and contract changes
Lifecycle and state transitions
Failure modes and recovery
Security / privacy / integrity implications
Observability and evidence produced
Testing and verification strategy
Migration / compatibility / rollback
Alternatives and tradeoffs
Open questions and decisions required
Implementation slices
Acceptance evidence
```

For Epiphany specifically, a design should additionally state:

- Which claims are observed, derived, provisional, and accepted.
- What is canonical versus rebuildable.
- What happens when a source is unavailable, malformed, stale, or unknown.
- Which relations are factual versus inferred continuity candidates.
- Which human acceptance event promotes a provisional claim.

That is how the documentation system reinforces the product’s epistemic discipline instead of being an unrelated bureaucracy.

## ADR template

ADRs should be short but more consistent than they are now:

```text
Status
Title
ID
Date
Owners / decision participants
Supersedes / superseded by
Related ADRs and designs

Context
Decision drivers
Decision
Invariants introduced
Consequences
Alternatives considered
Non-decisions
Reversibility / migration
Compliance and enforcement
Follow-up work
```

The key missing section in many ADR habits is **compliance and enforcement**. If a decision cannot identify how the project will detect violations, it risks becoming a historical essay rather than architecture.

## Examples and anti-examples

Create a dedicated, deliberately curated corpus:

```text
docs/standards/
  document-kinds/
    adr.md
    design.md
    epic.md
    story.md
    research.md
    process.md

  examples/
    story/
      good-schema-operation-registry.md
      bad-vague-validation-card.md
      good-small-adapter-law-card.md
      bad-hidden-design-in-story.md
    design/
      good-contract-first-verification.md
      bad-tool-list-without-decisions.md
    adr/
      good-authoritative-data-boundary.md
      bad-implementation-plan-pretending-to-be-adr.md
```

A bad example should be intentionally invalid and annotated:

```md
<!-- invalid because:
- no non-goals
- acceptance criteria are non-observable
- no verification plan
- no link to authoritative design
- “add validation” does not identify a boundary or invariant
-->
```

Do **not** use random old documents as “good examples” merely because they exist. Curate them against the template and label what they demonstrate. Existing documents can become candidates after review and normalization.

## Markdown checker: yes, dogfood it

This is exactly a good use of Epiphany’s Markdown parsing capability, but it should be built as two distinct things:

1. A narrow **documentation contract checker** needed now.
2. A broader document-analysis capability that may later become product functionality.

Do not block the process fix on a grand document-intelligence system. The first checker can operate on Flexmark’s parsed AST plus EDN/YAML frontmatter and return structured findings:

```clojure
{:document/path "docs/kanban/stories/engineering-assurance-..."
 :document/kind :story
 :result :invalid
 :findings
 [{:rule/id :story/missing-verification
   :severity :error
   :location {:line 42}
   :message "Story requires a Verification section."}

  {:rule/id :story/weak-acceptance-criterion
   :severity :warning
   :location {:line 58}
   :message "Criterion has no observable outcome or test reference."}]}
```

The checker should understand document kinds and relations, not just Markdown typography.

### Initial rules

**Errors:**

- Frontmatter fails the schema for its declared document kind.
- Required section is missing, duplicated, or out of allowed hierarchy.
- Story references a nonexistent design, epic, dependency, or ADR.
- A `ready` story is over five points, lacks acceptance criteria, lacks verification, or has unmet dependencies.
- An ADR marked approved has no decision, alternatives, consequences, or compliance section.
- A design that requires a decision links to no ADR or points to a draft/rejected ADR.
- A completed task lacks a completion-evidence record.
- A generated board snapshot is stale relative to source cards, once the board CLI is available.

**Warnings initially, later ratcheted:**

- Acceptance criterion is too vague: “works,” “is correct,” “tests pass,” “supports,” or “handles” with no named observable behavior.
- Empty or one-line non-goal, risk, verification, or failure-mode sections.
- Design contains implementation commitments without a corresponding interface/data contract section.
- A task references no source file, namespace, interface, or concrete affected boundary where such references are expected.
- An unchecked external Markdown link.
- Undefined acronym on first use.
- Heading skips levels or duplicate headings make sections ambiguous.
- A document has no inbound/outbound relation where its kind normally requires one.

**Never automatically fail solely because:**

- A section is short.
- Prose style is unusual.
- A design uses a table or diagram differently.
- An exception exists—but exceptions must be structured, dated, and reviewable.


## Stronger authoring process

The bulk-writing failure needs a process change too.

### Before writing a batch

1. Select the document kind and template.
2. Identify the authoritative source documents.
3. Identify one approved good exemplar and one anti-example.
4. Produce a per-document “evidence plan” before prose: required links, sections, invariants, tests, and non-goals.
5. Generate one document, lint it, review it, then use it as a constrained exemplar—not a loose pattern—for the rest.

### During writing

- Create **one story per implementation seam**, not one story per topic.
- Require a named owner for each invariant and verification claim.
- Disallow implied cross-card work.
- Use a structured `open-questions` field instead of silently guessing.
- Preserve an explicit “why this is not larger” statement for every card.


### Before board admission

- Run document contract checks.
- Run a relation check: ADR -> design -> card; card -> dependency; design -> research/decision input where required.
- Review against a human rubric.
- Only then transition `incoming -> accepted`.
- Only transition `accepted -> ready` when execution and verification are concrete enough for another person or agent to implement without reinterpreting the task.


## Existing cards need rework

The 10 ENG-017 cards should **not** be promoted from `incoming` as they are. They should become first-class test fixtures for the new standard:

- Keep the current versions as a **bad / incomplete batch-authoring baseline** in Git history, not as normative examples.
- Write the document-contract system first.
- Rewrite one card—probably ENG-017A—as the canonical good engineering-story exemplar.
- Use that approved exemplar and checker feedback to revise the remaining nine.
- Add a review note stating which ambiguity was removed from each rewrite.

That turns this mistake into useful training and regression material rather than silently replacing weak documentation with another untested style.

## Work to add

This deserves a **documentation-governance design**, likely preceding the assurance implementation lane:

1. **Documentation taxonomy and template contracts** — define kinds, metadata, sections, relations, statuses, and exception mechanism.
2. **Curated examples and anti-examples** — establish reviewed canonical samples per kind.
3. **Markdown document-contract checker** — parse frontmatter and Markdown AST, emit structured findings.
4. **Board admission gate** — enforce story completeness and relation integrity before `ready`.
5. **Documentation quality baseline migration** — classify existing corpus, normalize high-value active documents, record grandfathered exceptions.
6. **Batch-authoring protocol** — require exemplar-first, linted, reviewable batch generation.
7. **Optional semantic analysis phase** — detect vague acceptance criteria, contradictory claims, missing evidence links, and drift across related documents.

The documentation checker itself should follow ADR-004: schema-validated findings, example/anti-example fixtures, property tests for parser behavior, mutation tests around rule enforcement, and explicit separation between “structurally invalid” and “requires human judgment.” The existing verification architecture is a useful precedent for that separation, but it needs the new documentation standard to prevent it from drifting in turn.

The immediate next step should be a **small planning record or design for documentation governance**, not another bulk card-writing pass.
