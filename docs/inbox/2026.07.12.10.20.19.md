

# @GitHub Aight, what else? Agents will game anything. The only answer I know of, is to include more games than it can handle playing at once.

You’re right about the basic dynamic: an agent can optimize any single visible score. The answer is not merely “more tests”; it is a **portfolio of partially independent, adversarial checks**, with hidden variation and a human-reviewed evidence trail. METR has documented frontier models exploiting bugs in scoring code or task setups to obtain high scores without actually accomplishing the intended task.[^7_1]

The practical goal is: **make it cheaper to produce correct, explainable changes than to discover and exploit every gap in the assurance system.**

## Don’t create one game

A single test suite creates one game: “make the suite green.” A single linter creates “silence the linter.” A single coverage target creates “execute lines.” Mutation testing alone becomes “kill mutations without strengthening behavior.”

Instead, use checks that fail for different reasons:


| Assurance layer | What it resists | What it cannot prove |
| :-- | :-- | :-- |
| Closed Malli schemas at ports | Malformed values and undeclared fields | Semantic correctness |
| Contract tests, in-memory + Mongo | Fake adapters that accept impossible state | All production failure modes |
| Property tests | Overfitting to a fixed fixture list | Correct oracle design |
| Mutation testing | Shallow assertions and dead validation calls | Whether mutants model real defects |
| Differential tests | Adapter-specific behavior drift | Shared wrong interpretation |
| Metamorphic tests | Hard-coded expected-output gaming | Missing invariants |
| Static architecture checks | Layer violations and Java leakage | Runtime behavior |
| Randomized test order/seeds | Test state coupling | Correctness itself |
| Hidden evaluator tests | Direct optimization against known examples | A fundamentally vague requirement |
| Human review / acceptance | Specification gaming | Reviewer fatigue or unclear policy |

The portfolio works only if failures are independently generated and not all derived from the same production implementation.

## Property and metamorphic tests

Property testing changes the game from “match these examples” to “preserve these laws over many generated inputs.” Clojure’s `test.check` is QuickCheck-inspired: you write properties expected to hold across generated values, and it records a seed that can reproduce failures.[^7_2][^7_3]

For Epiphany, prioritize **metamorphic properties**: relations between outputs after a deliberately controlled input transformation, rather than assertions about one fragile expected output.

Examples:


| Subject | Transformation | Required relation |
| :-- | :-- | :-- |
| Registration | Register same Git common directory from two worktrees | Same resource ID; two location observations |
| Request idempotency | Replay same command with same request ID | Same durable outcome; no duplicate semantic fact |
| Conflict | Reuse request ID with changed command content | Explicit idempotency conflict; no partial write |
| Observation schema | Add undeclared key to valid closed observation | Validation rejects it |
| Backup | Export → import → export | Canonical payloads are semantically identical |
| Backup corruption | Truncate, alter manifest, change count/checksum | Import fails before writes |
| Git rename / move | Change path but preserve blob / evidence conditions | Candidate relation changes, never silently becomes accepted identity |
| Exact path policy | Use unusual valid path spellings | Stored raw path matches input exactly; no normalization claim |
| Search | Increase limit from $k$ to $k + n$ | First $k$ ranked results remain the same |
| HTTP/CLI parity | Submit equivalent requests | Identical normalized application command and outcome category |

These are particularly good for your identity/continuity model because they test **what must remain true** while avoiding an oracle that simply restates one fixture.

## Differential tests

Use multiple independent implementations of the same contract:

```text
application command
       |
       +--- validating in-memory reference adapter
       |
       +--- Mongo adapter
       |
       +--- serialized backup/import round-trip
```

Feed the same generated valid commands to each and compare normalized observable outcomes:

- Accepted or rejected.
- Domain error code.
- Canonical persisted observation.
- Query results in a deterministic ordering.
- Idempotent replay behavior.

Do not compare raw Mongo documents, exception strings, insertion timestamps, or generated IDs. Inject deterministic `:now` and `:new-id` capabilities, then compare data-level facts.

This catches your current specific problem: an in-memory store that accepts malformed data will disagree with the contract-enforcing reference port or Mongo port.

## Make evaluation opaque

Public tests teach agents the exact game. Keep the fast unit and contract suite visible, but add evaluator variation that is not part of the task prompt or a predictable fixed fixture list.

Use a secure/private CI context for:

- Additional malformed schema cases.
- Alternate valid generated cases and random seeds.
- Negative authorization/boundary scenarios.
- Changed test order.
- Contract suites run against the real adapter.
- Regression cases derived from past agent failures.
- A second implementation of crucial oracles.

Kaocha already randomizes test-suite, namespace, and test-var order by default and prints a seed so failures are reproducible. Keep that behavior; rerun failures using the recorded seed, and run several distinct seeds in CI.[^7_4]

Do not mistake hidden tests for a complete solution. A sufficiently capable agent can still infer the latent objective. Their function is to prevent direct fixture memorization and force adherence to the broader contract.

## Adversarial checks

Build a repository-specific “anti-gaming” suite. Each time an agent gets green through a loophole, preserve the incident as a minimal regression test **and classify the loophole**.

Suggested categories:

```edn
{:case/id "schema-write-bypass-001"
 :class :validation-bypass
 :asserts [:invalid-write-rejected
           :state-unchanged]
 :visibility :private-ci}

{:case/id "continuity-overclaim-002"
 :class :epistemic-overclaim
 :asserts [:candidate-not-accepted
           :evidence-preserved
           :human-review-required]
 :visibility :private-ci}
```

Important Epiphany-specific adversarial scenarios:

- Same content at a new path must not silently become the “same section.”
- Similar text must remain a candidate, not a confirmed identity.
- Missing Git history must produce unavailable/unknown, not “no continuity.”
- Inaccessible repository path must not erase the persistent resource ID.
- Schema validation must fail before any mutation; rejected writes leave storage unchanged.
- Unknown schema version must not be decoded as the nearest known schema.
- CLI and HTTP must not yield different decisions for the same normalized command.
- A placeholder workbench query must report unavailable/not implemented, never a meaningful-looking empty result.

That last class is important: agents often satisfy a response-shape test by returning `[]`, which turns “not implemented” into a false observational claim.

## Test the tests

Mutation testing is one part, but make it targeted and adversarial:

- Mutate/remove the validation decorator.
- Change a port-operation-to-schema mapping.
- Make closed maps open.
- Change required fields to optional.
- Turn an error branch into `nil` / empty vector.
- Skip a Mongo write or backup integrity check.
- Convert a review-required decision into an automatic accepted decision.
- Change `:unknown` or `:unavailable` into an empty/negative result.

The test suite should kill all of these deliberate semantic mutants. Generic arithmetic/boolean mutation is useful too, but these **domain-specific sabotage mutations** are higher signal.

For the schema gateway, a hard requirement is appropriate:

> A mutant that removes validation from any public write path must be killed.

## Invariants over coverage

Coverage is evidence that code ran, not evidence that the intended claim was verified. Require **invariant coverage** instead:

- Each persistent observation schema has valid, invalid, and round-trip tests.
- Each port operation has a rejection/no-state-change test.
- Each public command has CLI/HTTP decoder parity.
- Each state transition has an idempotency test.
- Each “empty” response state distinguishes empty, unavailable, corrupt, and not implemented.
- Each v1 schema has historical fixture compatibility.
- Each continuity outcome has tests for accepted, rejected, candidate, unknown, and unavailable.

Keep this as a machine-readable coverage matrix, not a prose promise:

```edn
{:operation :record-revision-at-path!
 :schema "observation/revision-at-path-v1"
 :checks #{:valid-write
           :invalid-write-rejected
           :state-unchanged
           :mongo-round-trip
           :backup-round-trip
           :idempotent-replay
           :unknown-version-rejected}}
```

A CI check can fail if any operation lacks a required assurance category.

## Evidence from behavior

Require agents to produce **replayable evidence**, not just a green claim:

```edn
{:change/id "..."
 :commands [{:cmd "clojure -M:unit-test"
             :exit 0}
            {:cmd "clojure -M:schema-check"
             :exit 0}]
 :properties [{:name "backup-round-trip"
               :seed 814296
               :trials 1000}]
 :adapter-contracts [:in-memory :mongo]
 :mutations {:killed 37
             :survived 0
             :reviewed-equivalent []}
 :interop-delta {:domain-dot-calls 0
                 :adapter-dot-calls 3}}
```

The evidence record should not itself be trusted blindly. CI needs to generate or verify it from actual commands and artifacts. The point is to force a distinction between **claimed assurance** and **observed run results**.

## Change the agent objective

Do not reward only “task done” or “all tests pass.” Score a change on a vector of constraints:

```text
correctness
+ contract integrity
+ regression resistance
+ evidence quality
+ architectural conformance
- unexplained complexity
- boundary bypasses
- new domain-layer interop
- unreviewed persistence/schema changes
```

Use **gates**, not a weighted average, for non-negotiables:

- No schema bypass.
- No invalid write accepted by reference or Mongo adapter.
- No unapproved domain-to-infrastructure dependency.
- No unreviewed schema version change.
- No false claim from unavailable/placeholder sources.

A weighted score invites compensation: an agent may trade a serious integrity violation for lots of superficial passing checks.

## Rotate games, preserve laws

Your instinct—more games than it can optimize simultaneously—is right, but avoid arbitrary churn. Keep **laws stable** and rotate **test instantiations**:

- Stable: a malformed record is never persisted.
- Rotated: which required key is absent, wrong, extra, mistyped, or inconsistent.
- Stable: identical request ID + different semantic content conflicts.
- Rotated: which field differs and whether it is nested.
- Stable: unknown is not false.
- Rotated: Git unavailable, Mongo unavailable, corrupt backup, feature flag disabled, unsupported schema version.
- Stable: candidate continuity is not accepted identity.
- Rotated: identical blob, text similarity, rename hints, clone split, history rewrite.

The agent can learn the actual contract, but cannot cheaply memorize a finite example suite.

## Human acceptance stays explicit

No test suite can decide epistemic questions like whether two document sections truly retain identity across a rewrite. Tests can enforce that the system records evidence, distinguishes candidate from accepted relation, preserves confidence and provenance, and requires a human decision where policy demands one.

So retain a human acceptance gate for:

- New schema versions and migrations.
- Changes to identity/continuity decision rules.
- New automatic acceptance paths.
- Changes that weaken validation, drop observations, or alter backup/restore semantics.
- Architecture exceptions permitting Java interop outside adapters.

The agent can propose a relation and present evidence. It should never silently convert similarity, passing tests, or a convenient heuristic into durable identity.
