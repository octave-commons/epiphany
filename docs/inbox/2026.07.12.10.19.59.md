

# @GitHub Aight whatever, I have clojure installed. Look over the code for me then for obvious issues.

I found several real issues worth addressing before treating the current green test count as a release signal. The most important are an incomplete backup/restore operation, unsafe parsing of HTTP EDN input, and persistence operations that are not crash-safe.

## High priority

| Issue | Why it matters | Recommended correction |
| :-- | :-- | :-- |
| **`restore-drill` is only an export stub** | `epiphany.domain.backup/restore-drill` exports a backup, prints a message, then returns `{:drill-status :export-complete}`. It does **not** drop data, re-import it, compare the re-export, or check inaccessible repositories, despite its docstring promising all of those stages. | Either implement the full drill or rename it to `export-for-restore-drill` and make it explicit that restore verification is not available. Add a test that fails unless drop, import, compare, and source-access checks occur. |
| **HTTP EDN bodies use `read-string`** | `epiphany.infra.http/read-body` passes externally supplied request text to `read-string` for `application/edn`. That reader is not an appropriate boundary parser for untrusted HTTP data; it can accept reader features and application-defined tagged-literal behavior that a data-only EDN parser should reject. | Require `clojure.edn` and replace it with `edn/read-string`; reject unsupported tags and return a 400 response on parse errors. Apply the same rule to the Lucene version-file read. |
| **Backups have no integrity or schema verification** | `import-from-file` checks only `:format`. It does not validate manifest version, expected collection keys, document counts, a checksum, or whether `:data` is structurally valid before handing it to the import port. A truncated or altered backup can therefore appear valid until data has already been changed. | Make the manifest contain a schema/version and SHA-256 of a canonical payload; validate all of it before importing. Import into temporary collections or use a transactional/replace strategy, then promote only after verification. |
| **Metadata writes are non-atomic** | Repository metadata and Lucene version metadata are written with `spit` directly to their destination. A process crash or disk interruption can leave a partial file; concurrent registration can race on the “missing metadata then write” path. | Write a fully flushed temporary file in the same directory, then atomically rename/move it; handle “target appeared” races by rereading and comparing identity. |

## HTTP boundary issues

The REST layer’s content negotiation and request validation are too permissive for a public or even shared-local API. `Accept` is selected by substring rather than media-type parsing, and query parameters are manually decoded without a defined policy for duplicate keys, malformed percent encoding, or limits.

More concretely:

- `:limit` is accepted directly from JSON and passed downstream without checking integer type, lower bound, or an upper cap. A negative, string, enormous, or otherwise malformed value can become a 500 error or a resource-exhaustion request rather than a clean 400.
- EDN/JSON parse failures appear to reach broad exception middleware, producing a 500 response instead of a client-error response. Invalid client input should consistently produce 400 with a stable problem type.
- `wrap-exceptions` returns exception messages to clients. For adapter failures, that can disclose filesystem paths, Git command details, Mongo details, or internal service topology.
- `text/plain` serialization uses `(str data)`, which is not a stable API representation and can expose implementation-oriented Clojure formatting.

I would add a small boundary validation layer: strict request schemas, `limit` constrained to a bounded integer range, explicit content-type handling, and an error mapping that logs detailed exceptions server-side while returning generic safe 500 details.

## Backup semantics

The backup module’s stated model is sensible—Git remains canonical for content and Mongo holds observations—but the operational contract is currently weaker than the documentation implies. The code includes `drop-all-collections!`, `inaccessible-sources`, export, and import helpers, but the orchestrating restore drill does not actually connect them.

There is also a data-model mismatch to investigate: `inaccessible-sources` reads `"repository-location"` from backup data, while the drop function names collections like `"repository-location-v1"`. If the exported map uses physical collection names, source accessibility checks silently inspect an empty list. Make logical backup keys and physical Mongo collection names separate, explicit mappings with a round-trip test.

## Continuity-model concern

For a system built around reviewable continuity rather than silent similarity-based identity, persistence and reconstruction must preserve **provenance** as carefully as it preserves observations. The current backup design appears to serialize all adapter data wholesale; that is acceptable only if restore preserves source fields such as observation timestamps, extractor version, Git revision/ref, review decisions, and request/idempotency identifiers without regenerating them.

I would test these invariants explicitly:

- Export followed by import produces a semantically identical observation set.
- Restore never creates a new repository resource ID for an already registered Git common directory.
- Rebuildable Lucene projections are excluded from authoritative backup claims and can be deterministically rebuilt.
- A missing repository path after restore stays an observed “unavailable source,” not a claim that the repository’s identity ceased to exist.
- A backup from an unknown future schema version fails before any destructive action.


## Product-correctness gaps

The workbench contains multiple “for now” placeholder paths: lineage, candidate/decision data, and inbox results are documented in the code as empty or placeholder responses. That is not necessarily wrong during active development, but it should be explicit in the interface rather than resemble completed functionality.

Use one of these policies:

- Hide unfinished routes/features.
- Return `501 Not Implemented` with a structured capability status.
- Return an explicit `{status: "unavailable", reason: ...}` result that cannot be mistaken for “there are no candidates” or “the inbox is empty.”

That distinction matters here: **empty evidence is a claim**, while **a feature not yet querying its source is an operational fact**. Conflating the two will make the knowledge system produce misleading continuity conclusions.

## Test gaps to add

The repository’s CI workflow is configured around the `:unit-test` alias, while `deps.edn` separately defines full and integration suites.

Prioritize tests for:

- Malicious/malformed EDN request bodies: rejected as 400, never evaluated.
- JSON body with `"limit": "20"`, `-1`, `0`, very large values, and non-finite numeric input.
- Truncated backup, wrong collection names, altered manifest count, unsupported future manifest version, and checksum mismatch.
- Restore-drill contract: assert all documented stages execute, or remove the claim.
- Simultaneous repository registration for the same Git common directory: one durable ID, no corrupted metadata.
- Metadata file interruption/race behavior using injected filesystem operations.
- Workbench endpoints: distinguish no results from unavailable/not implemented.

The source snapshot is at commit `6924b10`, whose commit message reports 528 tests and 1,336 assertions, but the issues above are mostly contract, boundary, and failure-mode cases that ordinary happy-path coverage often misses.
