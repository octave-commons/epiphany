## First: correct the test oracle

The in-memory observation adapter currently accepts almost any map:

```clojure
:record-ingestion-run! (fn [observation]
                         (swap! ingestion-runs conj observation)
                         nil)
```

The same pattern applies to checkpoints, section extractions, and revisions-at-path. It also does not validate required keys, UUID types, observation type, value shapes, foreign-key references, idempotency conflicts, or collection-level uniqueness. 

That creates a false green baseline: a command can produce malformed data that unit tests accept, then fail only when the Mongo BSON mapping or database constraints are involved.

### Define schemas once

Put the canonical map schemas in a neutral namespace—e.g. `epiphany.contract.observation`—that does **not** depend on Mongo, HTTP, Lucene, or CLI:

```clojure
(ns epiphany.contract.observation
  (:require [malli.core :as m]))

(def ResourceId [:uuid])
(def Oid [:string {:min 40 :max 64}])
(def Instant [:inst])

(def RepositoryLocation
  [:map {:closed true}
   [:observation/type [:= :repository/location-observed]]
   [:observation/id :uuid]
   [:observation/observed-at :inst]
   [:resource-id ResourceId]
   [:repository/path [:map {:closed true}
                      [:path/raw :string]]]])

(defn valid! [schema value]
  (when-not (m/validate schema value)
    (throw (ex-info "Invalid observation"
                    {:schema (m/form schema)
                     :explain (m/explain schema value)
                     :value value})))
  value)
```

Then both the application path and **every adapter** use the same validator:

```clojure
(defn record-repository-location!
  [store observation]
  (contract/valid! contract/RepositoryLocation observation)
  ;; Persist only validated data
  ...)
```

This makes invalid state unrepresentable at the port boundary, whether the port is in-memory or Mongo.

## Make doubles honest

Do **not** make the in-memory adapter a pretend Mongo implementation. Make it a **contract-enforcing reference adapter**:

- Validate every command input and persisted observation against the shared Malli schemas.
- Enforce declared invariants: immutable IDs, required foreign references, idempotency, and unique keys.
- Return the same domain-level errors as production—e.g. `{:code :idempotency-conflict}`—not Mongo driver exceptions.
- Keep its storage simple maps/atoms, but make behavior observable and deterministic.
- Explicitly document any intentional differences, such as no transaction or no query planner.

The current in-memory adapter says it is intended for unit tests and direct REPL use, but its `:record-*` functions merely append maps; that is the exact gap to close. 

A useful test matrix has three layers:

| Layer | Adapter | Purpose | Required gate |
|---|---|---|---|
| Domain | No adapter / pure functions | Validate rules and transformations | Fast, always |
| Contract | In-memory reference adapter | Enforce schemas and domain persistence invariants | Fast, always |
| Adapter integration | Ephemeral Mongo, Lucene, Git fixture | Prove BSON/index/Git behavior | Required in CI |
| End-to-end | CLI or HTTP against isolated services | Prove user-facing wiring | Required for release/merge paths |

The CI currently runs only `clojure -M:unit-test`; the repo defines separate full and integration aliases, so it is quite possible for CI to never exercise production persistence adapters. 

## Add static analysis

There is no checked-in static-analysis configuration in the repository root or near the source tree. Add `clj-kondo` first; it catches unresolved symbols, invalid arity, unused requires, suspicious bindings, and many idiomatic Clojure mistakes without pretending Clojure is fully statically typed.

## Use type checking selectively

Clojure does not need pervasive Java-like type annotation. For this project, the best leverage comes from **data schemas at boundaries** and **typed contracts around side-effecting ports**, rather than trying to impose static types on every pure domain function.

Adopt a layered policy:

- **Malli schemas** for domain entities, commands, events/observations, API input/output, and adapter port contracts.
- **Malli instrumentation in development and test** so functions fail at the wrong boundary, not later in Mongo or Lucene.
- **Clojure Spec or Malli generators** for property testing important laws: idempotency, ordering, lineage invariants, round trips, and serialization.
- **Typed Clojure only as a pilot**, perhaps for the Mongo document mapper boundary, after the contract vocabulary stabilizes. Do not make Typed Clojure a project-wide prerequisite; it will create friction in dynamic infrastructure code before it offers commensurate safety.

For example, instrument ports in dev/test:

```clojure
(require '[malli.dev :as dev])

(dev/start!
 {:report (dev/-multifn
           {:default dev/-fail!})})
```

The exact API choice may differ with your chosen Malli version, but the policy is what matters: schemas should run in test and development, and production validation should remain at all external input and persistence boundaries.

## Measure Java interop accurately

A grep count is a useful architectural signal but not a runtime call count. It tells you **where the representation leaks**, while runtime instrumentation tells you **where work is spent**.

Track both.

### Structural interop budget

Start with a CI report that counts direct interop only in non-adapter namespaces:

- Dot calls such as `(.getTime date)`.
- Constructor/static calls such as `(java.util.UUID/randomUUID)`.
- Java imports/type hints.
- `clojure.java.shell/sh`.
- Namespace-level `:import` use.

Treat these as warnings at first, then put a ratchet on them: no new direct Java interop in `domain`, `application`, `law`, or `shape` without an architecture exception.

Suggested policy:

| Namespace area | Direct Java interop | Rule |
|---|---:|---|
| `epiphany.domain` | 0 target | Inject clocks/ID/digest services; work with normalized data |
| `epiphany.application` | 0 target | Orchestrate ports, never JVM APIs |
| `epiphany.law` / contracts | 0 target | Describe data and port behavior only |
| `epiphany.infra.http` / CLI | Very low | Translation only; no business rules |
| `epiphany.infra.adapters.*` | Allowed | Adapter-specific Java is confined here |
| `epiphany.infra.git` | Allowed but narrow | Ideally wrap command/JGit behavior behind a small port |

The numbers you found show this needs a namespace gate, not a blanket “no Java” stance: Mongo, Lucene, and Git adapters legitimately bridge Java libraries, while HTTP and CLI should remain shallow composition layers. 

### Runtime interop measurement

For **runtime counts**, do not try to count arbitrary JVM method calls from Clojure at the language level. Instead, instrument the seams you own:

```clojure
(defn counted
  [metrics operation f]
  (fn [& args]
    (let [started (System/nanoTime)]
      (try
        (apply f args)
        (finally
          (swap! metrics update-in [:calls operation] (fnil inc 0))
          (swap! metrics update-in [:nanos operation]
                 (fnil + 0) (- (System/nanoTime) started)))))))
```

Wrap every adapter port function—Mongo read/write/query, Lucene index/search, Git commands, Ollama embedding, filesystem metadata operations—not every Java method. This measures the architecture-relevant question:

> How often does a user operation cross from functional/domain code into a JVM-backed infrastructure adapter, and how expensive is that crossing?

Expose a report after test or benchmark execution:

```clojure
{:calls {:mongo/record-checkpoint 42
         :lucene/search 10
         :git/read-blob 318}
 :nanos {:mongo/record-checkpoint 30120000
         :lucene/search 9012300
         :git/read-blob 80003200}}
```

If you truly need JVM-wide profiling, use Java Flight Recorder (JFR) or async-profiler for a focused run. But that answers performance questions; it will not identify conceptual leakage as cleanly as a structural budget plus explicit port metrics.

## Remove Java assumptions from domain code

The direct `java.util.Date` and UUID calls in domain namespaces are small individually but create the “two assumption systems” you described: the code is nominally functional and data-oriented, but its rules are tied to mutable JVM-era values and methods. The scan found examples across continuity, ingestion, review, lineage, concept, and export code. 

Use ordinary data and injected capabilities:

```clojure
;; Before
{:observation/id (java.util.UUID/randomUUID)
 :observation/observed-at (java.util.Date.)}

;; After
(defn observed
  [{:keys [new-id now]} command]
  {:observation/id (new-id)
   :observation/observed-at (now)})
```

At the composition root:

```clojure
{:new-id random-uuid
 :now #(java.time.Instant/now)}
```

For dates, standardize on `java.time.Instant` at infrastructure boundaries, or use a portable representation such as RFC 3339 strings / epoch milliseconds in domain data. Avoid `java.util.Date` as a domain type. This improves testing, serialization, temporal reasoning, and legibility.

## Untangle CLI and HTTP

Both interfaces should be thin adapters over a shared application command/query layer. Right now the HTTP namespace has parsing, content negotiation, exception mapping, profile resolution, and handler orchestration; the main CLI namespace also combines parsing, setup, profile behavior, Git commands, lifecycle handling, workbench startup, and output/exit behavior. That makes both difficult to test and easy to let drift. 

Refactor toward this shape:

```text
CLI argv -> CLI decoder -> application command -> application service -> ports
HTTP request -> HTTP decoder -> same command -> application service -> ports
                                                  |
                                        domain rules / contracts
```

Create a command vocabulary made only of data:

```clojure
{:command/type :repository/register
 :request/id "..."
 :repository/path "/path/to/repo"}

{:query/type :search
 :query/text "identity after rename"
 :search/mode :hybrid
 :limit 20}
```

Then write:

- `decode-cli`: argv to validated command map.
- `decode-http`: Ring request to the **same** validated command map.
- `execute`: application command to domain result.
- `encode-cli`: domain result/problem to text and exit status.
- `encode-http`: domain result/problem to JSON and HTTP status.

This gives you one semantic command contract, one validation path, and two presentation adapters. It also means CLI/HTTP parity tests become simple: given equivalent input, both decoders must produce the same command; given the same result, both encoders must preserve the same outcome category.

## Practical sequence

1. Add shared Malli schemas for every persisted observation and every public command.
2. Make the in-memory adapter validate and enforce the same port invariants as Mongo.
3. Add Mongo integration tests using an ephemeral service; make them a required CI job.
4. Add `clj-kondo` and establish a clean baseline before ratcheting warnings to errors.
5. Generate and commit an interop inventory; enforce zero new direct Java interop in domain/application namespaces.
6. Introduce injected `:now`, `:new-id`, hashing, and filesystem/Git capabilities; migrate `Date` and UUID creation out of domain functions.
7. Extract shared application commands from the existing CLI and HTTP adapters.
8. Add boundary property tests, adapter contract tests, and CLI/HTTP parity tests.
9. Instrument adapter port crossings and publish counts/timing per integration suite.
10. Only then refactor the high-density Mongo/Lucene code, where isolation will make the Java-heavy code acceptable and contained.

The goal is not “eliminate Java.” It is to make Java libraries an explicit infrastructure implementation detail, while the domain and application layers remain a single, inspectable system of data, contracts, evidence, and reviewable decisions.
