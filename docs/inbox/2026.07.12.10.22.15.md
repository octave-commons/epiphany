## Governance model

I suggest we explicitly establish three layers:

```text
Constitution
  PROCESS.md
  "What must remain true as process changes"

Legislation
  docs/process/*.md
  "Current policies and workflows that implement the constitution"

Operations
  AGENTS.md, docs/kanban/AGENTS.md, templates, tooling, CI
  "How contributors and agents execute current policy"
```

An ADR is not above the constitution. It is the authoritative record for an architectural decision **within** the constitutional constraints; it must not override epistemic integrity, evidence preservation, or explicit acceptance requirements.

## Role of PROCESS.md

`PROCESS.md` should be stable, short enough to hold in working memory, and difficult to change casually. It should avoid naming:

- A specific Kanban CLI.
- A specific board directory layout.
- Exact board states and transitions.
- Current test commands.
- Current WIP limits.
- Exact document templates.
- Current tool selections such as a linter or mutation engine.

Those are legislative or operational details that should be able to evolve.

Instead, it should establish rules such as:

1. **Work is accountable.** Material implementation work is traceable to a bounded work item before it is represented as complete.
2. **Claims retain their epistemic tier.** Sources, derived findings, proposals, and accepted decisions are never conflated.
3. **No irreversible or architectural change without appropriate evidence and explicit decision.**
4. **Work is sliced to preserve reviewability, testability, and recoverability.**
5. **Acceptance is evidence-based.** A completion claim requires recorded observed evidence, not self-attestation.
6. **Process is adaptive but not silently mutable.** Rules may change through a documented proposal, bounded trial, review, and explicit adoption/reversion.
7. **Exceptions are explicit and expiring.** An exception identifies its scope, owner, rationale, risk, review date, and follow-up.
8. **The system improves from failures.** Escaped defects, ambiguous tasks, invalid research, and agent gaming attempts become durable evidence for improving standards, examples, and checks.

That is constitutional: it says what a future Kanban workflow, document checker, or CI policy must honor without forcing one implementation indefinitely.

## Role of docs/process

`docs/process/` becomes the place for changeable, specific policies. Your existing Kanban document belongs there.

Suggested future structure:

```text
docs/process/
  README.md
  kanban.md
  research.md
  document-governance.md
  decision-making.md
  review-and-acceptance.md
  exceptions.md
  governance-change.md
```

Each legislative process document should declare:

```yaml
---
kind: process-policy
status: draft | active | superseded | retired
implements: ["PROCESS.md#work-is-accountable"]
owner: ...
review-cycle: quarterly
supersedes: []
superseded-by: null
---
```

That lets the constitution remain stable while process policies become deliberately revisable.

## Relationship to AGENTS.md

`AGENTS.md` should become an **operational navigation document**, not another constitution. It can state:

```text
Before changing code:
1. Identify the governing ADR/design/card.
2. Follow the current process policy.
3. Run the required local checks.
4. Record observed evidence on the work item.
5. Escalate ambiguity rather than inventing policy.
```

It should link outward to the authoritative source rather than copy large policy blocks. Right now it is valuable but overloaded: it contains local service assumptions, architecture constraints, quality policy, board directions, and commands in one place.

## Reframing Kanban

Your intent for `docs/process/kanban.md` is sound:

> Once work reaches the board, this document governs how it is triaged, scoped, moved, executed, reviewed, handed off, and completed.

That means its specific state machine, WIP behavior, provenance/ledger requirements, and CLI workflow are **allowed to change**—but each must implement constitutional rules such as accountable work, explicit evidence, and reviewable transition history.

The one thing we need to resolve in the rewrite is whether `docs/kanban/AGENTS.md` is:

- The **tool-specific companion** to `docs/process/kanban.md`, containing current `eta-mu` command usage and board layout; or
- A duplicate policy document that should be consolidated into `docs/process/kanban.md`.

My preference is:

```text
docs/process/kanban.md    policy: workflow, states, gates, exceptions
docs/kanban/AGENTS.md     execution: CLI commands, file conventions, tool behavior
docs/kanban/board.json    generated view: never normative
```

That keeps “why/when” separate from “which command.”

## Research belongs here

Your insight about research is not ancillary; it is a constitutional gap.

`PROCESS.md` should say that work requiring novel factual, technical, or design claims must be grounded in evidence proportional to consequence. But it should not define the exact research-report frontmatter or source-record format. Those belong in:

```text
docs/process/research.md
docs/standards/document-kinds/research-report.md
docs/standards/document-kinds/source-record.md
```

Constitutional rule:

> Research is a bounded activity that reduces a named uncertainty through recorded method and evidence; its outputs distinguish observed sources, derived findings, provisional recommendations, and accepted decisions.

The legislative research process then defines the actual artifact types, templates, review thresholds, source-quality rules, and checker behavior.

### Pass one: constitutional skeleton

```md
- Kanban
- Research
- Documentation governance
- Review and acceptance
- Exceptions
- Engineering standards

```


### Pass two: legislative alignment

After the constitution is accepted:

1. Rework `docs/process/kanban.md` so it explicitly implements constitutional clauses.
2. Replace imported `STYLE.md` with an Epiphany-specific engineering-policy document.
3. Write `docs/process/research.md`.
4. Create document-kind templates, examples, anti-examples, and checker rules.
5. Trim `AGENTS.md` into a current operational entry point with links rather than duplicated policy.

## The central distinction

The rewrite should enforce one idea everywhere:

> **Constitutional rules protect meaning; legislative policies organize changing practice; operational documents tell a person or agent what to do today.**

That division gives you the flexibility you want without allowing drift to become silent contradiction.

The consequential question for drafting `PROCESS.md` is: **should the constitution impose hard universal gates—for example, “no accepted decision without recorded evidence”—or should it state only principles and leave all gates to legislative process documents?**
