## The symbols are bindings

Read it as a small declarative DSL:


| Symbol | Binding in your process language |
| :-- | :-- |
| `η` | The field of experienced effects without a settled source explanation |
| `Discovery` | The operation that inventories, records, and orients to η |
| `Describe` | State the intended meaning/problem in human terms |
| `specify` | Make outcomes, constraints, and failure conditions observable |
| `define` | Declare laws, schemas, contracts, error shapes, and admissibility |
| `μ` | The field of producible effects whose explanation/limits are incomplete |
| `shape` | Pure value transformation under law |
| `extern` | Boundary that invokes and decodes foreign capability |
| `domain` | Pure Epiphany meaning and decisions |
| `infra` | Effect orchestration and adapter composition |
| `Π` | The continuously integrated product/result of this cycle |

That is internally coherent. The parenthesized groups establish **two distinct intermediate products**:

```clojure
(-> Describe specify define)
;; => semantic / contractual basis

(-> μ shape extern domain infra)
;; => implemented operational capability
```

The first does not become the second automatically. It constrains and gives meaning to the second.

## One syntax adjustment

If you want it to look like ordinary Clojure and communicate its evaluation shape more directly, I would write:

```clojure
(η Discovery
   (-> Describe
       specify
       define
       (-> μ
           shape
           extern
           domain
           infra))
   Π)
```

But this subtly makes `μ` appear to be an input to the output of `define`. That may be correct—`define` creates laws that constrain the productive field—but it risks hiding the fact that η and μ are independent fields.

Your original grouped form better preserves the relation:

```clojure
(η Discovery
   (-> (-> Describe specify define)
       (-> μ shape extern domain infra))
   Π)
```

I would call the inner groups **forms**, not phases:

```text
η-form: encounter and discover

Contract form: describe -> specify -> define

Construction form: μ -> shape -> extern -> domain -> infra

Π-form: integrate the constrained construction into product
```


## What “continuous integration” means here

Π is not a final terminal state. It is the current integrated product of a cycle, and it becomes part of the next η.

```clojure
(-> Π
    observed-effects
    η)
```

Meaning: the product is deployed, tested, read, used, questioned, and fails in ways that generate newly experienced effects. That feedback closes the loop:

```text
η
  -> discovery
  -> contracts
  -> construction
  -> Π
  -> newly encountered η
```

That is why the anomaly rule is not an aside. It is the process’s evaluation behavior when a form yields an unexpected value.

## The key law implied

The actual invariant remains:

> A value may not cross from one form into the next without acquiring the shape, contract, provenance, and verification appropriate to that boundary.

In code terms:

```text
η observation
  cannot directly become a domain conclusion

μ capability
  cannot directly become infrastructure behavior trusted by the product

foreign value
  cannot directly become domain data

green check
  cannot directly become accepted completion
```

This connects your process constitution, research policy, Kanban policy, and engineering kernel without making them all the same document.
