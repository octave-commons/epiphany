

# @GitHub Where does splint fall in to this? Or Mutation testing?

**Splint and mutation testing complement schema enforcement; neither replaces it.** Use schemas to reject invalid data at boundaries, Splint to prevent confusing or risky code shapes, and mutation testing to prove your tests actually detect broken behavior.[^6_1][^6_2]

## Where Splint fits

Splint is a Clojure linter focused on **style and code shape**, in the same general space as Kibit/RuboCop-style feedback; it is not a schema validator or static type checker.[^6_3][^6_1]

For Epiphany, its value is primarily architectural hygiene around the areas you identified as difficult to reason about:

- HTTP and CLI translation code.
- Java-interop-heavy adapter code.
- Nested `cond` / `let` / `try` blocks.
- Redundant or indirect Clojure forms that obscure data flow.
- Large functions whose shape suggests they mix decoding, application orchestration, infrastructure access, and presentation.

It should run alongside clj-kondo, not instead of it. clj-kondo is the baseline static analyzer for unresolved symbols, invalid arities, unused bindings, namespace problems, macro-aware analysis, and project-specific lint configuration.[^6_4]


| Tool / mechanism | Main question answered | Best enforcement point |
| :-- | :-- | :-- |
| Malli law schemas | “Is this command, observation, result, or document structurally valid?” | HTTP/CLI decode, port writes, port reads, backup import |
| clj-kondo | “Is this valid, maintainable Clojure code?” | Every PR and local editor feedback |
| Splint | “Is this code written in a clear, idiomatic, simple shape?” | Every PR initially as warnings; later ratchet selected rules |
| Adapter contract tests | “Do in-memory and Mongo obey the same observable contract?” | Required unit + integration CI |
| Mutation testing | “Would the tests fail if important behavior changed?” | Nightly, pre-release, focused changed namespaces |
| JFR / port metrics | “Where does runtime actually cross into infrastructure/Java?” | Benchmark and integration runs |

## What Splint should enforce

Avoid treating its default rule set as universal truth. Establish an Epiphany-specific `.splint.edn` and turn rules on only where they support a local architectural policy; Splint supports project-root configuration through `.splint.edn`.[^6_5]

Good initial targets:

- Flag excessively nested conditionals and long function bodies in `infra.http` and `infra.main`.
- Flag needless anonymous functions / indirect invocation patterns that obscure the path from request to command to port.
- Flag complexity-prone transformations in HTTP and CLI code, motivating extraction into pure decoder/encoder functions.
- Encourage small `case`-based command dispatch over sprawling nested `cond` routing, when it makes the command vocabulary clearer.
- Identify expressions that make the domain code look Java-shaped rather than data-shaped.

Do **not** use Splint as a blunt “remove all Java interop” tool. Mongo, Lucene, and JGit adapters are where Java interop belongs; it should be isolated, named, and tested there. The enforcement policy belongs in a separate custom check: direct Java calls should be prohibited or tightly budgeted in `domain`, `application`, and `law`, but allowed in `infra.adapters.*`.

## Add Splint now

Add a dev alias and a project config. The currently published Splint artifact is `io.github.noahtheduke/splint`, with a `1.20.0` release listed on Clojars.[^6_6]

```clojure
;; deps.edn
:aliases
{:splint
 {:extra-deps {io.github.noahtheduke/splint
               {:mvn/version "1.20.0"}}
  :main-opts ["-m" "splint.main" "src" "test" "dev"]}}
```

```edn
;; .splint.edn
{:paths ["src" "test" "dev"]

 ;; Begin by reporting. Promote the selected rules only after
 ;; the baseline has been reviewed and intentionally cleaned up.
 :config
 {:enabled true}}
```

Then:

```bash
clojure -M:splint
clojure -M:lint
clojure -M:cljfmt
```

The exact config keys and CLI invocation should be verified against the pinned Splint version before committing; its configuration is intentionally controlled from `.splint.edn`.[^6_7][^6_5]

## Custom structural checks

The highest-value checks for this repository are project-specific and go beyond Splint’s generic idiom rules.

### Enforce layer boundaries

Write a small CI checker—using clj-kondo analysis output, rewrite-clj, or a simple initially conservative namespace scan—that fails if:

- `epiphany.domain.*` imports Java classes or uses direct dot interop.
- `epiphany.application.*` requires `epiphany.infra.*`.
- `epiphany.law.*` requires either `domain` or `infra`.
- `epiphany.infra.http` calls Mongo/Lucene/Git directly rather than an application port.
- `epiphany.infra.main` performs domain logic rather than argv decoding, composition, execution, and output encoding.
- A port write is exposed without a schema mapping and validation wrapper.

That is much closer to the failure mode you described than generic style linting: it catches the moment two representational systems get mixed without an explicit adapter.

### Maintain an interop ledger

Generate a checked-in `reports/interop.edn` or CI artifact with namespace-level counts:

```edn
{:generated-at "..."
 :src/epiphany/domain {:dot-calls 0
                       :constructors 0
                       :type-hints 0}
 :src/epiphany/application {:dot-calls 0}
 :src/epiphany/infra/adapters/mongo {:dot-calls 238}
 :src/epiphany/infra/adapters/lucene {:dot-calls 65}}
```

Set a ratchet:

- No new domain/application/law interop.
- Adapter totals may rise only in a PR that explains why and adds an adapter-level test.
- Overall counts are informational initially, then budgeted after the refactor baseline stabilizes.

This turns “Java use looks ugly and confuses agents” into a reviewable, measurable architectural constraint.

## Where mutation testing fits

Mutation testing changes source behavior—e.g. flips a comparison, changes a branch, removes a function call—and reruns tests. If tests remain green, that mutant “survived,” showing that the suite did not prove the affected behavior. Mutant describes this directly: it alters namespaces and reports mutations that leave the test suite passing.[^6_2]

It is especially valuable after you enforce schemas, because it catches **shallow schema tests** such as:

```clojure
(is (registry/valid? "observation/ingestion-run-v1" good-record))
```

That test proves only that a known-good fixture passes. It might not fail if:

- The production write path stops calling `validate!`.
- A required schema field becomes optional.
- The schema mapping accidentally points an ingestion operation at the wrong record schema.
- The in-memory adapter ignores the validation wrapper.
- Validation errors are caught and the invalid record is stored anyway.

The mutation you want killed is not merely “a predicate changed.” It is “the enforcement boundary vanished and invalid data persisted.”

## Mutation tests to prioritize

Run mutation testing only on **pure and boundary-critical namespaces**, not the entire project at first.

Start with:

- `epiphany.application.validate` or whichever validation gateway you add.
- `epiphany.law.registry`.
- Schema-to-port-operation mapping code.
- In-memory observations adapter.
- Mongo encoder/decoder functions.
- HTTP and CLI decoders.
- Continuity, lineage, evidence, and review decision functions.
- Backup manifest validation and restore planning.

Do not begin with:

- Full Mongo/Lucene/JGit adapter internals.
- `infra.main` process lifecycle code.
- Workbench rendering.
- Network and Docker orchestration.

Those areas will generate expensive, noisy mutants and are better covered by contract/integration tests plus port metrics.

## A useful mutation score

Use a scoped score and track survivors by category:

$$
\text{mutation score} =
\frac{\text{killed mutants}}
{\text{killed mutants} + \text{surviving mutants}}
$$

Do not set an arbitrary global 100 percent requirement. Equivalent mutants exist—changes that alter syntax but not behavior—and dynamic Clojure code can produce low-value mutation noise.

Instead, require:

- **0 surviving mutants** in the schema gateway and schema-operation mapping.
- **0 surviving mutants** that remove a port-boundary validation call.
- A reviewed justification for equivalent/irrelevant survivors.
- An improving score in pure domain namespaces.
- Mutation testing as nightly or merge-queue work, not necessarily every local edit.


## Tool choice

The older Clojure tool **Mutant** is a mutation-testing library that mutates source namespaces and reruns tests, but its documented workflow is Leiningen-oriented (`lein trampoline mutate`). That is a poor fit for this `deps.edn`/Clojure CLI project unless you are willing to add Leiningen only for mutation runs.[^6_2]

A more promising current direction is **Heretic**, presented as fast, practical, integrated Clojure mutation testing, with its project documenting mutation operators. Validate its current Clojure CLI integration and maintenance status in a short spike before making it a required CI dependency.[^6_8][^6_9]

I would make the decision through a one-week proof:

1. Run each candidate on `epiphany.application.validate` and `epiphany.law.registry`.
2. Measure setup complexity, runtime, mutation report quality, and false/equivalent-mutant rate.
3. Verify it runs under `clojure -M` without changing the project to Leiningen.
4. Require that it identifies a deliberately removed `validate!` call as a surviving mutant.
5. Keep the tool only if developers can act on its survivors.

## CI placement

```yaml
jobs:
  static:
    steps:
      - run: clojure -M:lint
      - run: clojure -M:splint
      - run: clojure -M:cljfmt

  contract:
    steps:
      - run: clojure -M:unit-test
      - run: clojure -M:schema-check

  integration:
    services:
      mongo:
        image: mongo:8
        ports: ["27017:27017"]
    steps:
      - run: clojure -M:integration-test

  mutation:
    if: github.event_name == 'schedule' || github.event_name == 'workflow_dispatch'
    steps:
      - run: clojure -M:mutation --namespaces epiphany.application.validate,epiphany.law.registry
```

Splint creates **pressure toward readable shapes**. Schema gates establish **what data is allowed to cross a boundary**. Mutation testing establishes whether the tests would notice if either protection were removed. Together, they address both the ugly implementation surface and the false confidence of a permissive in-memory test world.
