**Epiphany** is a local-first, Git-backed knowledge archaeology system for turning years of notes, documentation, code, and project history into a searchable, evidence-based research environment.

Its central purpose is not simply “chat with my files.” It is to help you answer questions like:

- *What have I already written or thought about this?*
- *When did this idea first appear?*
- *How did it change over time?*
- *Which notes genuinely continue the same line of thought, and which only look similar?*
- *What contradictions, gaps, abandoned questions, or unfinished research threads remain?*
- *What should I research next, and what evidence led to that suggestion?*

## Core idea

Epiphany treats your local Git repositories as a long-lived personal and technical archive.

It scans repository history—not only the files currently checked out—and extracts Markdown documents into structured, revision-scoped evidence units such as:

- documents;
- headings and sections;
- paragraphs, lists, tables, quotes, and code blocks;
- front matter;
- links;
- named entities;
- commit and path history.

It then makes those units searchable through several complementary methods:

- lexical/full-text search;
- semantic/vector retrieval;
- metadata and date filtering;
- path and repository filtering;
- Git history traversal;
- continuity signals;
- later, graph relationships and geospatial data.

The goal is fast retrieval, but with the ability to inspect exactly *why* a result or relationship exists.

## Evidence first

The project is designed around a strict distinction between four kinds of knowledge:

| Category | Meaning |
|---|---|
| Observed | Directly present in Git history, a file, a commit, a path, or another canonical source |
| Provisional | A model, heuristic, or similarity process proposed a possible relationship |
| Accepted | A human explicitly reviewed and accepted an interpretation |
| Rejected | A human explicitly rejected a candidate; it remains auditable but is hidden by default |

That distinction is fundamental.

For example, Git can establish that:

```text
At commit X, path docs/identity.md contained blob Y.
```

Epiphany may infer:

```text
This 2025 section probably refines a section from 2023.
```

But the latter is not promoted to fact merely because an embedding model, n-gram model, LLM, or graph algorithm found it plausible.

## Git identity model

Epiphany is designed for real local repositories: moved folders, clones, mirrors, worktrees, history rewrites, renamed files, copies, and long-lived paths whose purpose changes.

A registered repository receives a minimal stable identity record in its Git metadata:

```clojure
{:resource-id #uuid "..."}
```

That allows the system to recognize the same local repository after it moves on disk.

The platform keeps evolving operational metadata elsewhere, including:

- observed locations;
- repository instances;
- shared-history families;
- ingestion state;
- projection checkpoints;
- extraction records;
- cached/indexed representations;
- review decisions;
- lineage candidates;
- history-rewrite observations.

A repository’s identity is therefore not its current path.

```text
/home/me/notes
  -> moved to
/archive/personal-notes
  -> still the same Epiphany resource
```

## Continuity model

Epiphany does not assume that a path equals an idea.

A file can stay at the same path while completely changing purpose. Conversely, a file can move or be copied while retaining its exact content. Git does not permanently store semantic “rename” or “same document” facts; it stores object history, while rename detection is a comparison-based inference.

So Epiphany keeps separate relationships for:

- **same path across adjacent commits** — observed path continuity;
- **same blob at another path** — exact content relocation or copy;
- **similar text or structure** — inferred similarity;
- **same concept or ongoing argument** — provisional or accepted semantic continuity;
- **epoch boundary** — a reviewable claim that a long-lived path was repurposed.

For Markdown, continuity can be estimated from signals such as:

- text similarity;
- front matter stability;
- overlap in explicit links;
- time gaps;
- named-entity overlap.

For code, later policies can use different evidence:

- AST-shape overlap;
- public-symbol overlap;
- dependency changes;
- test overlap;
- comments and documentation;
- code-specific structural changes.

Markdown and code are intentionally not forced into one generic continuity model.

## Phase 1

Phase 1 concentrates on local Git repositories and Markdown, because that is enough to solve the immediate problem: excavating a large personal corpus of notes and documents.

The first useful loop is:

```text
Register repository
  -> ingest Git/Markdown history
  -> extract revision-scoped sections
  -> search current and old expressions
  -> inspect exact historical evidence
  -> compare revisions or passages
  -> follow a dated lineage
  -> review candidate connections
  -> create a research question
  -> export an evidence packet
```

A successful Phase 1 should let you select a present-day note, find older relevant expressions, inspect their exact commits and source spans, and decide whether they genuinely belong to the same intellectual thread.

## Search and retrieval

Epiphany is not vector-search-only.

Vectors are useful for finding related material when wording has changed, but they are only one ranking signal. A good answer may depend on literal wording, dates, paths, metadata, Git ancestry, links, named entities, and reviewed relationships.

The intended retrieval strategy is hybrid:

```text
Lexical relevance
+ semantic similarity
+ path and source scope
+ dates and commit history
+ metadata/front matter
+ continuity features
+ graph relationships
+ human-reviewed links
= ranked evidence candidates
```

That makes it possible to ask both broad and precise questions:

```text
"What did I write about identity and repository moves?"
```

```text
"Show me documents under .ημ/ from 2023–2025 that discuss event sourcing."
```

```text
"What earlier note most directly led to this current architecture decision?"
```

```text
"Where did this line of thought become unresolved?"
```

## Interfaces

The project exposes one primary executable:

```bash
epiphany
```

with a supported short form:

```bash
ep
```

Examples:

```bash
ep resource register ~/notes
ep ingest request res_...
ep search "repository continuity"
ep evidence show sec_...
ep compare sec_... sec_...
ep lineage show sec_...
ep review decide cand_... accept
ep question create "When did this idea become explicit?" --evidence sec_...
```

The interface strategy is layered:

| Interface | Role |
|---|---|
| CLI | Scripts, automation, bootstrap, diagnostics, repair, reproducible work |
| TUI: `ep tui` | Keyboard-first search, evidence triage, comparison, lineage, review, operations |
| Browser GUI | Rich Markdown reading, visual timelines, diffs, health views, later graphs/maps |
| Native desktop shell | Deferred; only justified if an offline bundled workstation becomes important |

The CLI and TUI may work directly against local application services. The browser GUI uses the REST API. All interfaces use the same command/query contracts and must not bypass the evidence and domain layers by talking directly to storage systems.

## Infrastructure direction

Epiphany is intended to run across your local networked machines, with work divided by capability rather than pretending every node is equivalent.

The broad architecture includes:

- Clojure services for ingestion, orchestration, domain logic, and interfaces;
- Git object access for canonical repository history;
- MongoDB for durable operational records, events, user review decisions, and projections;
- Elasticsearch for full-text, vector, metadata, and eventually geospatial retrieval;
- Neo4j or an equivalent graph store for explicit entity/relation traversal;
- append-only events for replayable state changes and rebuildable projections;
- disposable caches for high-speed access;
- local inference for embeddings, taxonomy prototyping, entity extraction, and evidence-grounded QA.

The powerful GPU machine is best used for inference, embedding, reranking, and heavier extraction. The stronger CPU machine can host primary search/orchestration work. The weaker machines can handle routing, cold storage, backups, monitoring, replication support, and low-priority jobs.

## Longer-term direction

Once the Git/Markdown archaeology loop works, Epiphany can expand into a broader local research platform:

- AST extraction for Clojure, ClojureScript, Go, Python, C, Lua, Ruby, Bash, YAML, EDN, and JSON;
- syntax-tree n-grams and code-aware retrieval;
- knowledge graph construction;
- user-created labels and concepts;
- LLM-assisted taxonomy proposals;
- external scientific datasets;
- geopolitics and geospatial analysis;
- research planning;
- evidence-backed question answering;
- tool-using agents that retrieve and analyze before reasoning.

But the core remains the same:

> Epiphany is an evidence-preserving system for recovering and extending the intellectual history already embedded in your repositories.

It is meant to turn accumulated notes from an opaque pile into a navigable memory: searchable, reviewable, historically grounded, and useful for deciding what to think about next.
