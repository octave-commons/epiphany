# @GitHub octave-commons/epiphany clone this, set up java and clojure, get ready to test this

You’ve already got the repo cloned in the sandbox; next steps are to install Java + Clojure tooling, then mirror whatever the project’s CI uses to run tests (via `deps.edn`, `tests.edn`, and `.github/workflows/test.yml`).

## Repo clone

On your own machine or in the sandbox, the clone step is:

```bash
git clone https://github.com/octave-commons/epiphany.git
cd epiphany
```

In the sandbox, the repository now exists under `epiphany/` with key files like `deps.edn`, `tests.edn`, `bin/epiphany`, and `test/epiphany/integration_suite_test.clj`.

## Install Java (JDK)

Epiphany is a Clojure project, so you need a JVM; OpenJDK 17 is a good default on Debian 13.

```bash
sudo apt-get update -y
sudo apt-get install -y openjdk-17-jdk
```

You can verify Java with:

```bash
java -version
javac -version
```

Those commands ensure Clojure can run and any tooling that relies on the JVM will work consistently.

## Install Clojure CLI tools

This repo is configured with `deps.edn`, which means it expects the Clojure CLI (`clojure`) rather than Leiningen by default.

On Debian-style systems, the standard install script is:

```bash
curl -O https://download.clojure.org/install/linux-install-1.11.1.1409.sh
chmod +x linux-install-1.11.1.1409.sh
sudo ./linux-install-1.11.1.1409.sh
```

After that, confirm:

```bash
clojure -M -e '(println "Clojure OK")'
```

This gives you the `clojure` command that uses `deps.edn` aliases and paths defined in the project.

## (Optional) Install Leiningen

The project is CLI-first with `deps.edn`, but if you want Leiningen as well (for any dev convenience), you can add it:

```bash
curl https://raw.githubusercontent.com/technomancy/leiningen/stable/bin/lein -o /tmp/lein
chmod +x /tmp/lein
sudo mv /tmp/lein /usr/local/bin/lein
lein version
```

Leiningen isn’t required by anything visible in `deps.edn` or `tests.edn`, but having it around doesn’t hurt for Clojure workflows.

## Project layout relevant to tests

From the cloned tree:

- `deps.edn` – Clojure CLI dependency + alias config; this is where test aliases are likely declared.
- `tests.edn` – test runner / alias wiring for how suites should be invoked.
- `test/epiphany/integration_suite_test.clj` – at least one integration test namespace under `test/`.
- `.github/workflows/test.yml` – GitHub Actions workflow that encodes the canonical “how CI runs tests”; you should mirror that locally.

There is also a `bin/epiphany` script which looks like the main entrypoint wrapper for the CLI/TUI; it’s useful to inspect when you want to exercise the app as part of integration runs.

## Discovering the test command

To “get ready to test”, the safest path is to follow whatever CI does and to read the test config files:

1. Open `.github/workflows/test.yml` and look for the step that runs tests (likely a `run: clojure -M:test` or similar).
2. Open `deps.edn` to identify the aliases (e.g. `:test`, `:integration`, etc.).
3. Open `tests.edn` to see how suites are grouped; some Clojure projects use this for test runner configuration (e.g. Kaocha, Cognitect test-runner).

Once you see the aliases and runner, you’ll know the precise command (for example, typical patterns include `clojure -M:test` or `clojure -X:test/run`, but use what the files actually specify).

## Running tests in your environment or the sandbox

After Java and Clojure CLI are installed and you’re in the repo:

1. Export a sensible JVM heap if needed:

```bash
export JVM_OPTS="-Xms256m -Xmx2g"
```

2. Run the same command CI uses (example pattern, adjust to what `test.yml` + `deps.edn` say):

```bash
clojure -M:test
```

3. To run only the integration suite, use the alias or namespace pattern configured in `tests.edn`, which includes `test/epiphany/integration_suite_test.clj`.

If anything fails, the first place to check is that the Clojure CLI and JDK versions match what CI uses, and that environment variables referenced in `test.yml` are set in your shell.
